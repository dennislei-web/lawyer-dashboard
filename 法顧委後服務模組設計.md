# 法顧「委後服務」模組設計規劃

> 依據同仁整理的《法顧部門服務SOP》（Google Docs）+ 現有 advisor 儀表板結構規劃。
> 日期：2026-06-11 · 狀態：設計討論稿（尚未實作）

---

## 1. 背景與缺口分析

現有 `public/advisor/` 儀表板覆蓋的是 **獲客側**：

| 現有 Tab | 對應 SOP 階段 | 資料來源 |
|---|---|---|
| 總覽 / 業務·律師 | 成案結果統計 | `advisor_cases`（Sheet 每日同步） |
| 跟進中案件 | 進線 → 聯絡 → 諮詢 → 報價 → 簽約（委前 pipeline） | `advisor_pending_cases`（輪值表同步） |
| 客戶 360 | 回流 / LTV / 續委任觸發 | `advisor_cases` 歸戶 |
| （對帳） | 儲值 / 扣抵現金流 | `advisor_transactions`（CRM 爬蟲） |

SOP 中 **簽約之後** 的整段流程目前完全沒有系統承接：

```
簽約付款 → 開案交接（業務→主管律師→分案）→ 委後服務（時數消耗/每月報告/回覆SLA）
        → 合約週期管理（時數預警/到期續約）→ 續委任 or 流失
```

對應 SOP 章節：
- **法顧業務SOP**：簽約（指定律師註記）、收款（印花稅）、交接事項準備、系統開案注意事項、開卷、開案群組回報、每月底檢查
- **法顧律師服務SOP**：三天內回覆、時數預估/即將超過/已超過 三段例稿、當天寫時數、每月報告、黏著度標籤（每月報告/換律師）、黃律師雙週會議
- **法顧行政SOP**：每月例行（已成案歸戶、上月業績總額、業務人別分類）→ 大部分已被現有 Sheet 同步自動化

## 2. 整體架構決策

1. **不採用 Next.js**。維持專案慣例：單檔 HTML tab + CDN Supabase JS，整合進 `public/advisor/index.html` 新增 tab（或拆 `public/advisor/service/` 子頁，視單檔行數成長決定；現有 2,327 行，預估加 1,500 行內可留在同檔）。
2. **新表是「儀表板原生」SoT，可直接在 UI 編輯** —— 與現有三張 Sheet 同步表（清表重灌）嚴格分離，避免使用者輸入被每日 sync 洗掉。關聯方式：以 `client_name`（+ 既有 `advisor_cases.id` 選填外鍵）軟連結。
3. **沿用既有 Auth / `lawyers` 表**，不另建 profiles。`lawyers.role` 已有 `admin / manager / lawyer`，法顧業務（克威/柏凱/偉志）以 `lawyers` 既有帳號對應，不需「第一位註冊者當 manager」的邏輯（那是綠地專案的規則，此處不適用）。
4. **RLS 改為列級權限**（現有 advisor 表是「登入可讀、admin 可寫」，新表需要業務可寫自己的案件）：
   - 業務：可讀寫 `owner_id = auth.uid()` 的服務案件
   - manager / admin：全部可讀寫
   - 沿用既有 `is_admin()`，新增 `is_manager()`（security definer，避免 RLS 遞迴）

## 3. 委後生命週期（pipeline）

接續 `advisor_pending_cases.current_stage`（…→ 提報方案 → 已成案）之後：

```
已簽約 → 收款確認 → 交接中 → 已分案 → 服務中 → 續約評估 → 已續約 ↻ / 已流失
```

| 階段 | 進入條件（SOP 依據） | 色票建議 |
|---|---|---|
| 已簽約 | 委任契約簽回（Dottedsign 或紙本） | blue |
| 收款確認 | 匯款水單 / 末五碼確認、收據+印花稅處理完 | teal |
| 交接中 | 開案交接表已建立、開卷群組已回報 | orange |
| 已分案 | 承辦主管完成分案、承辦律師打招呼 | purple |
| 服務中 | 服務期間內（時數消耗進行中） | green |
| 續約評估 | 合約到期前 60 天 或 時數用罄前（自動觸發） | gold |
| 已續約 | 產生新一期合約（回寫 advisor_cases 續委任）| green |
| 已流失 | 到期未續 / 中途解約 | red |

「已成交」口徑 = 已簽約以後全部階段（供業績統計，與既有 `advisor_cases` 成案口徑互相驗證）。

## 4. 頁面設計（advisor 儀表板新增 2 個 tab）

### Tab 5「委後服務」（主畫面）

**KPI 列（5 卡）**
1. 服務中客戶數
2. 本月時數消耗 / 異常數（超過預估的客戶數，red）
3. 60 天內到期合約數（gold）
4. 交接中案件數（停留 >3 天標紅 —— SOP「隔天會議、三天內回覆」精神）
5. 本期續約率（已續 / 應續）

**區塊 A：交接看板**（已簽約→已分案 的案件，橫向 4 欄 kanban 或表格+階段色票）
- 欄位：當事人 / 負責業務 / 簽約日 / 已收款? / 指定律師? / 交接停留天數
- 點入 → 案件詳情 drawer

**區塊 B：時數雷達**（服務中客戶）
- 表格：客戶 / 承辦律師 / 已購時數(點數) / 已耗用 / 剩餘% / 本月消耗 / 狀態標籤
- 狀態標籤對應 SOP 三段例稿：`正常`(green) / `即將超過`(orange, 剩餘<20%) / `已超過`(red)
- 「每月報告」「換律師」黏著度標籤（SOP 個案黏著度上標籤）

**區塊 C：合約到期雷達**
- 來源：`advisor_transactions.contract_end_date` + 手動維護
- 90/60/30 天三段提醒，到期未續自動轉「續約評估」

### 案件詳情（drawer 或 modal，非獨立路由）

1. **基本資料表單**（可編輯）：當事人、統編、聯繫管道(LINE@)、指定律師、承辦律師、管轄、相牽連案件、對造名稱（SOP：每份新合約要鍵入對造防利衝）
2. **開案交接 checklist** —— 直接把 SOP「系統填寫開案注意事項」結構化成勾選＋填值：
   - 契約簽回（電子/紙本）、收款齊、收據+印花稅、利衝建檔、對造已鍵入
   - 急件註記、已知庭期/時效屆日、雲端資料夾連結、會議記錄/錄音存放、印章、紙本卷所在
   - 全部勾完 → 可推進「已分案」
3. **跟追歷程 timeline**（`advisor_case_events`）：手動新增進度紀錄（通話、會議、時數回報、續約洽談…），對應 SOP「約通話後 line 上註記：承剛剛電話討論…」的留痕習慣
4. 刪除案件（僅 manager/admin）

### Tab 6「每月例行」（行政 SOP 自動化檢查表，第二期再做）

把行政 SOP 的人工月結變成系統檢查：上月成案歸戶筆數 vs Sheet 總額對帳（reconcile 區塊已有雛形可複用）、各業務人別分類統計、「諮詢卡片簽約狀態待更新」清單、每月時數表回報狀態（每客戶一勾）。

## 5. 資料表設計（草案 DDL 摘要）

```sql
-- 角色判斷（security definer 防 RLS 遞迴）
CREATE FUNCTION is_manager() RETURNS boolean
  LANGUAGE sql SECURITY DEFINER STABLE AS $$
    SELECT EXISTS (SELECT 1 FROM lawyers
      WHERE auth_user_id = auth.uid() AND role IN ('manager','admin'));
$$;

-- 1. 委後服務案件（儀表板原生 SoT，可編輯）
CREATE TABLE advisor_service_cases (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  advisor_case_id UUID REFERENCES advisor_cases(id),   -- 軟連結成案紀錄（可空）
  client_name     TEXT NOT NULL,
  client_vat      TEXT,                                -- 統編，歸戶 key 之一
  opposing_party  TEXT,                                -- 對造（利衝）
  stage           TEXT NOT NULL DEFAULT '已簽約' CHECK (stage IN
    ('已簽約','收款確認','交接中','已分案','服務中','續約評估','已續約','已流失')),
  designated_lawyer TEXT,                              -- 指定律師
  handling_lawyer   TEXT,                              -- 承辦律師
  owner_id        UUID REFERENCES lawyers(id),         -- 負責業務
  office          TEXT,
  line_channel    TEXT,                                -- LINE@ 連結
  fee_amount      NUMERIC(12,0),
  contract_start  DATE,
  contract_end    DATE,
  purchased_hours NUMERIC(8,1),                        -- 已購時數/點數
  monthly_report  BOOLEAN DEFAULT false,               -- 黏著度標籤：每月報告
  switch_lawyer   BOOLEAN DEFAULT false,               -- 黏著度標籤：換律師
  handover        JSONB DEFAULT '{}',                  -- 開案交接 checklist（鍵值勾選）
  note            TEXT,
  external_ref    TEXT,                                -- 預留 CRM 系統案件編號
  created_at      TIMESTAMPTZ DEFAULT now(),
  updated_at      TIMESTAMPTZ DEFAULT now()            -- trigger 自動更新
);

-- 2. 跟追歷程
CREATE TABLE advisor_case_events (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id     UUID NOT NULL REFERENCES advisor_service_cases(id) ON DELETE CASCADE,
  author_id   UUID REFERENCES lawyers(id),
  event_type  TEXT DEFAULT 'note',     -- note/call/meeting/hours/renewal/stage_change
  content     TEXT NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 3. 時數帳（手動 + 之後可由 CRM 爬蟲補）
CREATE TABLE advisor_hour_entries (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id     UUID NOT NULL REFERENCES advisor_service_cases(id) ON DELETE CASCADE,
  work_date   DATE NOT NULL,
  hours       NUMERIC(5,1) NOT NULL,
  entry_kind  TEXT DEFAULT 'work' CHECK (entry_kind IN ('work','quote')),  -- 工作/報價時數（SOP 區分）
  description TEXT,                    -- 工作事項（SOP：要寫得讓客戶覺得有價值）
  lawyer_name TEXT,
  created_by  UUID REFERENCES lawyers(id),
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

**RLS**（三表同模式）：
- SELECT：登入即可讀（跨業務透明，主管會議要看全貌；若之後要收緊再改 owner 限定）
- INSERT/UPDATE：`owner_id = (select id from lawyers where auth_user_id = auth.uid())` OR `is_manager()`
- DELETE：僅 `is_manager()`

**下拉選項**：來源沿用既有 `case_category` 六分類；委任項目用 SOP 案由自由文字＋常用清單（法律顧問、合約審閱、勞資、商務…）。金額顯示沿用既有 `fmtMoney`（NT$ 無小數）。

## 6. 資料來源與自動化

| 資料 | 來源 | 機制 |
|---|---|---|
| 新簽約案件 | 「跟進中案件」轉入 | UI 按鈕「轉入委後」帶入 client_name/業務/金額（一鍵建檔，不動 Sheet） |
| 合約迄日 / 儲值點數 | `advisor_transactions`（CRM 爬蟲已有 `contract_end_date`,`point`,`total_point`） | nightly script 比對 client_name/統編回填，UI 可覆寫 |
| 時數消耗 | 北所 Google 試算表 / 各分所 CRM 系統 | 第一期手動輸入；第二期評估擴充既有爬蟲抓 CRM 時數 |
| 續委任成案 | `advisor_cases`（Sheet 同步） | 「已續約」案件與新成案紀錄互相驗證（reconcile 區塊模式） |

## 7. 分期 Roadmap

**Phase 1 — 交接 + 合約管理（建議先做，資料現成）**
migration（3 表 + is_manager + trigger）→ 委後服務 tab：交接看板 + 合約到期雷達 + 案件詳情（表單/checklist/timeline）→「跟進中案件」加「轉入委後」按鈕

**Phase 2 — 時數管理** ✅（2026-06-12 完成）
時數雷達 + 手動時數輸入 + 三段預警標籤；advisor_transactions 點數自動對照。
實作發現：CRM 點數單位＝分鐘（60 點 = 1 小時），`total_point` 為當前剩餘快照 →
已耗 = Σpoint − total_point，全自動、免手動記時數；手動時數帳保留給北所（試算表記錄）fallback。
合約迄日：案件欄位優先、CRM `contract_end_date` fallback（⚡ 標記）；modal 可一鍵帶入。

**Phase 3 — 每月例行 + 服務品質** ✅（2026-06-12 完成）
每月例行 tab：上月業績歸戶（業務人別+總額，對應行政 SOP 月結）、每月報告/時數表勾稽
（「標記已寄」寫入 monthly_report 事件）、互動健康（依跟追歷程算距上次互動，>14 橘 >30 紅）、
黃律師雙週會議摘要產生器（時數預警/到期/換律師/未寄報告/無互動，一鍵複製）。

## 8. 待確認問題（實作前需要你或同仁拍板）

1. **時數的 SoT**：各分所記錄方式不一（北所 Google 試算表、其他所用系統）。第一期手動輸入可行，但長期要不要爬 CRM 時數模組？
2. **歸戶 key**：客戶用 `client_name` 還是統編？同名公司/個人戶會撞，建議統編優先、姓名 fallback。
3. **業務帳號**：克威/柏凱/偉志目前在 `lawyers` 表有沒有對應 Auth 帳號？（owner_id 綁定的前提）
4. **編輯權限範圍**：時數要不要開放承辦律師自己填（SOP 要求律師當天寫時數）？若要，律師角色也需要寫入權限。
5. **與輪值表 Sheet 的關係**：簽約後該筆在 Sheet 標「已成案」即停止追蹤，委後資料只活在儀表板 —— 同仁是否接受雙軌（Sheet 管委前、儀表板管委後）？
