# 合署律師上架調整清單

> 一位律師「資深轉合署」或「新進司法官合署」時，儀表板要動的所有地方。
> 維護者：改完用 `grep -rn '某現有律師名'`（如 `柯雪莉`）反查還有沒有漏的硬編碼點。

---

## 0. 先懂兩種機制（決定要不要改 code）

| 機制 | 哪些頁靠它 | 要不要改 code |
|---|---|---|
| **動態**：CRM 案件 `group_name` 含「合署」就自動算合署（`isPartner()` = `group_name.indexOf('合署')`） | 營收(revenue)、OKR、五年推估、法顧(advisor)、財務(finance)、諮詢分析(consultation) | **免改**。前提：①CRM 端把案件 group 設成「北所合署（XX律師）」②對帳/案件爬蟲有撈到 |
| **寫死名單**：分潤 cohort 儀表板、歷史 owner 歸因生效日、律師權威名單、forecast 可轉池 | partners 合署律師頁、firm_amount 歸因、forecast | **要手動改 + 有些要上傳 Excel** |

⚠️ 關鍵陷阱：動態機制讓新合署律師的「金額」自動進營收，**看起來沒事**，但分潤頁、歷史拆帳、可轉池這些寫死的地方會默默漏掉或算錯。

## 2. 完整清單（依系統分組）

### A. 對帳 / 案件 ETL ✅ 已根治
- `scripts/scrape_reconciliation.py` — 原 `ALL_GROUPS` 寫死部門清單已移除，改抓所有部門。**新合署 group 自動納入，免改。**

- `scripts/group_inference.py` `OFFICE_DEFAULT` — 只在開「新據點」時要加，新律師免改。

### B. 合署生效日 `PARTNER_SINCE`（⚠️ 三份副本要同步）
轉合署前的案件算「純所內實收」、之後算「合署」，靠生效日切。**三個檔各有一份、目前內容不一致**：
1. `scripts/backfill_partner_attribution.py:48`（只有 4 人 — 嚴重落後）→ 控 `revenue_records.firm_amount`
2. `scripts/build_case_cost_data.py:32`（18 人，有蘇萱無李家泓）→ 控成本分析
3. `public/revenue/index.html:~3955`（有蘇萱無李家泓）→ 控前端歷史 owner 判斷
- **每加一位都要三份一起加** `'名字': 'YYYY-MM-01'`（轉合署當月 1 號）。

### C. 律師 / 部門權威名單 & 非諮詢排除
- `public/revenue/index.html` `WEBSITE_PARTNERS`(3645)、`WEBSITE_LAWYERS`(3657)、`PARTNER_OFFICE`(3969 只非台北所要加)
- `scripts/build_case_cost_data.py` `LAWYERS_BY_OFFICE`(54)、`NON_CONSULTING_LAWYERS`(72)
- `scripts/build_lawyer_tenure.py` `LAWYERS_BY_OFFICE`(34)、`ALREADY_PARTNER`(49)、`MANAGERS`(56)

### D. 分潤 Cohort 儀表板（最重，且要 Excel）
這是 partners 合署律師頁的專屬分潤/拆帳矩陣，**全寫死且依賴 Excel 分潤表**：

## 3. 兩種情境的精簡 SOP

### 情境 A：資深律師轉合署
1. **CRM 端**先把該律師案件 group 設成「北所合署（XX律師）」（動態機制的根）
2. B 三份 `PARTNER_SINCE` 加生效日
3. C `WEBSITE_PARTNERS` / `WEBSITE_LAWYERS`（+ 非台北所才加 `PARTNER_OFFICE`）
4. DB 補 `lawyers.partner_terms`
5. 分潤 Excel 上傳 Drive → D 全套名單/顏色（drive_client、parse_senior、build_embedded）+ 人數字串
6. forecast `ALREADY_PARTNER`（從可轉池移除）
7. 重跑：scrape_reconciliation → backfill_partner_attribution → build_case_cost_data → partners sync

### 情境 B：新進司法官合署
同 A，但分潤走 judicial 線：`parse_judicial.py`（正則 + 自案縮寫）、`build_embedded.py` `JUDICIAL_*`、跨轉分析 E 的 `JUDICIAL` set。司法官通常無「轉合署前純律師期」，`PARTNER_SINCE` 用其加入日。

## 4. 建議（治本）

現在 20+ 處散落、已證實會漏（李家泓）。建議收斂成**單一 source of truth**：
- 在 DB `lawyers` 表用既有欄位（`partner_terms` + 新增 `partner_since` / `cohort_type` / `display_color`）當唯一名單；
- 各 script / 前端開頁時讀 DB 生成名單，不再各自寫死；
- 保留分潤 Excel 流程（那是外部資料），但 cohort 歸屬、顏色、生效日、人數一律 DB 驅動。

如此「上架一位合署律師」= 在帳號管理頁填一列 + 上傳 Excel，免改 code。
